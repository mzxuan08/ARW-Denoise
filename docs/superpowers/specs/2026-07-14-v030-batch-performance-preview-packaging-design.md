# ARW Denoise 0.3.0 批处理、性能、预览与发布优化设计

日期：2026-07-14  
基线：ARW Denoise 0.2.0 / PMRID ONNX FP16 / Sony A7C II  
发布策略：四项优化全部完成后只发布一个 0.3.0，不生成中间用户版本。

## 1. 目标与范围

0.3.0 同时完成四个目标：

1. 为长队列提供可理解的进度、ETA 和真正及时的取消。
2. 降低整图处理的 RAM/VRAM 峰值和总耗时，不改变 DNG 安全性。
3. 增加源 ARW 与降噪 DNG 的本地对比预览。
4. 在保留完整离线 GPU 能力的前提下精简发布内容并提供更高压缩率的交付物。

不在本版本范围内：更换 PMRID 模型、训练 A7C II 专用模型、修改支持机型白名单、引入云端服务、增加照片管理或调色功能。

## 2. 方案选择

采用“在现有稳定架构上逐层优化”的方案。保留 PySide6 GUI、PMRID ONNX 会话、现有 RAW 后处理与 dnglab 发布路径，以可测试的独立组件增加任务控制、性能工作区、预览服务和发布剖面。

不采用 GUI/队列整体重写，因为这会同时增加持久化恢复和 DNG 发布回归风险。也不采用只修改界面与压缩方式的外围方案，因为它无法解决当前照片不能及时取消和整图内存副本过多的根本问题。

## 3. 组件边界

### 3.1 任务控制器

新的任务控制器只负责取消信号、阶段与子进度、耗时样本和状态快照。它不知道 Qt，不直接访问 SQLite，也不写 DNG。解码、每个 GPU/CPU tile、后处理块和 dnglab 过程边界都检查同一个取消令牌。

取消是显式的非错误结果。点击取消后，当前任务尽快停止，对应 dnglab 子进程被定向终止，仅删除本任务的 UUID 临时文件。已发布 DNG 和所有源 ARW 不受影响。SQLite 记录进入 `cancelled`，可在以后重新入队。

### 3.2 进度与 ETA

进度事件包含任务 ID、阶段、已完成单元、总单元和单调时间戳。阶段权重由真实基准数据生成，不在 GUI 中写死。GUI 显示当前文件、阶段进度、整队列进度、已用时间和预计剩余时间。

ETA 使用当前运行中最近完成照片的滚动中位数，按实际引擎和像素数归一化。少于两个有效样本、引擎发生回退或当前文件尺寸偏差过大时显示“计算中”，不伪造精确数字。

### 3.3 低内存处理工作区

现有算法的数值规则保持不变。优化通过复用浮点缓冲区、就地剪裁/融合、及时释放已不可达的整图数组和为后处理提供预分配输出完成。不使用降低精度、缩小分辨率或绕过均值/高光保护的方式换取内存。

在改变缓冲区所有权前先建立分阶段耗时、进程 RAM 峰值和 NVIDIA VRAM 峰值采集。优化后的两张 A7C II 基准以相同模型、tile 和设置对比 0.2.0，中位总耗时至少降低 15%，RAM 峰值至少降低 25%。若某项经分阶段数据证明受外部 dnglab/I/O 硬限制，可以基于证据调整数值目标，但不能放宽画质和安全验收条件。

GPU OOM 仍按 `1024 → 768 → 512 → 384 → 256` 的有限序列回退。用户取消比 OOM 重试优先，一旦取消不得再创建下一个 tile 会话或转入 CPU。

### 3.4 对比预览服务

预览不参与正式 DNG 生成。它使用 LibRaw/rawpy 对源 ARW 和已完成 DNG 应用相同的快速本地显影配置，产生仅用于界面的 8-bit sRGB 预览。两边必须使用相同的白平衡、曝光、色彩空间、旋转和尺寸，避免预览管线伪造差异。

预览界面为独立窗口，不挤压主队列。支持适应窗口、100%、同步缩放、同步拖动和可拖动的左右分割线。仅对已完成任务开放预览。缓存键包含源/输出路径、文件大小、修改时间、显影版本和预览尺寸；任一项改变即重建。预览失败只显示可复制错误并允许重试，不改变任务完成状态。

在 A7C II 整尺寸文件上，冷启动的适应窗口预览目标在 5 秒内显示，命中缓存后目标在 1 秒内显示。预览生成始终在工作线程中运行，不阻塞主界面或照片队列。

### 3.5 发布剖面

构建过程从可执行文件的实际导入和离线 CUDA 自检结果确定必需 Qt、ONNX Runtime 和 NVIDIA DLL，移除未被使用的 Qt 插件、翻译、ONNX 工具与 TensorRT provider。不依赖开发机 PATH 删除任何 CUDA DLL；每次精简后都必须在清空环境中通过真实 CUDA 推理。

最终同时交付兼容 ZIP 和固实高压缩包。两者解压后的相对路径集合和每个文件 SHA-256 完全一致，并共用一份受版本控制的发布清单。ZIP 仍为无额外解压工具的首选，固实包用于尽可能降低下载体积。

0.3.0 兼容 ZIP 不得大于 0.2.0 的 1.84 GB 基线；固实包的目标不超过 1.50 GB，且至少比同版 ZIP 小 15%。如果 NVIDIA 必需可再分发 DLL 的熵限制使 1.50 GB 不可达，必须保留“至少小 15%”的相对目标并记录各目录压缩贡献，不得删除离线自检所需 DLL。

## 4. 错误处理与恢复

- `cancelled` 与 `failed` 保持独立；取消不触发错误弹窗，失败不伪装成取消。
- 突发退出后，原有运行态任务仍恢复到等待；明确的用户取消保留为已取消。
- 取消与 OOM 同时发生时以取消为最终状态，不回退 CPU。
- dnglab 子进程超时、取消和普通失败使用同一个定向回收路径，并验证临时文件不再增长。
- 预览缓存损坏或过期时原子重建；不将损坏预览显示给用户。
- 构建精简如果导致任一离线自检失败，发布立即失败，不产生降级的“可能能用”包。

## 5. 数据与兼容性

SQLite 任务表增加当前阶段、阶段进度、总进度、已用时间、峰值 RAM/VRAM 和取消时间字段。所有迁移保持幂等，0.2.0 数据库启动后不丢失队列、输出路径、参数快照或历史结果。

任务建立时仍固定引擎模式和高级参数快照；修改全局设置不改变旧任务。预览只保存可重建缓存，不将 RGB 图像存入 DNG 或任务数据库。

## 6. 测试设计

### 6.1 单元与组件测试

- 取消令牌在解码、tile、后处理和 dnglab 边界可重现地终止。
- 进度事件单调、不超过总量，阶段切换后整体进度不倒退。
- ETA 在样本不足、异常值、引擎切换和像素数变化时符合显示规则。
- OOM 和取消竞争不会启动新 tile 或 CPU 回退。
- 临时文件清理仅命中当前任务的 UUID 路径。
- 新旧后处理路径对固定输入的输出相等或每样本相差不超过 1 个 RAW 码值。
- 预览两边共用显影配置，同步缩放/平移坐标变换可逆，缓存键变化会失效。
- 0.2.0 SQLite 固定样本可重复迁移并保留字段。
- ZIP 与固实包的解压清单与文件哈希完全相等。

### 6.2 集成与真实验收

1. 运行全部无 GPU 单元测试与真实 CUDA 自检。
2. 在 RTX 3070 Ti Laptop 上重复处理 `DSC01959.ARW` 和 `DSC01928.ARW`，采集每阶段耗时、RAM/VRAM 峰值、tile 和输出哈希。
3. 与 0.2.0 输出逐像素对比，检查 CFA、黑白电平、裁切、色彩元数据和高光保护。
4. 在解码、GPU tile、后处理和 DNG 写入四个时点人工注入取消，确认响应和清理。典型 GPU tile 取消延迟目标小于 2 秒，硬上限为 10 秒。
5. 运行至少 150 项的持久化队列压力测试，覆盖失败隔离、立即取消、重试、强制退出恢复和重名输出。
6. 对两张已完成 DNG 生成预览，检查缩放、拖动、分割、错误恢复和缓存失效。
7. 将 ZIP 和固实包分别解压到新用户目录，清空 `PATH`、`PYTHONPATH`、模型与 dnglab 环境变量，执行 CUDA 真实推理和 GUI 启动测试。

## 7. 发布门槛

0.3.0 只在以下条件全部满足后交付：

- 全部自动测试通过，150 项队列压力测试通过。
- 两张 A7C II 真实 DNG 可重新打开，数值、元数据和高光保护无回归。
- 取消不损坏源文件、已发布 DNG 或其他任务，不留下可增长的临时文件。
- 性能和内存指标达到目标，或对外部硬限制提供可重现的分阶段证据。
- 预览失败不影响正式处理，预览不上传任何数据。
- 两种发布包内容一致，均可完全离线使用，并附发布清单与 SHA-256。
